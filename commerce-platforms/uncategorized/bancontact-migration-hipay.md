Bancontact Integration Migration - HiPay

* Bancontact Integration Migration: What You Need to Know As part of the optimization of our payment infrastructure and to offer you an ever-richer service, HiPay is consolidating the acquisition of its Bancontact and Bancontact QR Code flows. From today until December 31, 2026 , all flows passing through our legacy partner will be progressively migrated to our new partner . The legacy integration will be permanently decommissioned on this date. Important Note: After December 31, 2026, all Bancontact flows will be operated exclusively via the new integration. ## TECHNICAL IMPACT SUMMARY The goal of this migration is to be as seamless as possible for your system. However, a minor update on your end is required for the classic Bancontact flow: Payment Phase Technical Impact Details Transaction Creation (Order API) Minor impact (Modification) For the classic Bancontact flow, the payment_product parameter value must be updated. URL and authentication remain strictly identical. API Response (Synchronous Response) No impact The format and fields of the API response from HiPay to your server do not change. Server-to-Server Notifications (S2S) Minor impact (Modification & Addition) Notifications for statuses 116 (Authorized) and 118 (Captured) will contain restructured and additional information sent by the new partner (IBAN, BIC, 3DS data) for both Classic and QR Code flows. ## DETAILS BY PAYMENT METHOD ### 1. BANCONTACT (CLASSIC REDIRECTION) #### A. Order API Impact (Transaction Creation) To initiate a classic Bancontact transaction via the new integration, you must update the payment_product parameter in your Order API request. The legacy value bcmc-mobile is deprecated and must be replaced by bancontact . #### B. S2S Notification Impact (Data Restructuring) With the legacy integration, notifications returned basic card information. With the new integration, the payload is enriched with detailed bank information and 3DSecure status. Warning: Ensure your JSON parser or HTTP request processing system accepts these new fields and structures without generating a validation error. Data Field Group Legacy Structure New Structure Notes payment_method[...] [card_expiry_month] [card_expiry_year] [issuer] [pan] New [bic_debtor] New [iban_debtor] [pan] New [token] Card expiry details have been moved to the parameters block below. payment_product_parameters Not present New Escaped JSON String containing: iban , bic , cardbin , cardExpiryMonth , cardExpiryYear ... This field is delivered as an escaped JSON string containing comprehensive banking details. three_d_secure[...] Not present New [authentication_message] New [authentication_status] New [authentication_token] New [eci] New [xid] Full visibility on the 3D Secure authentication flow. ### 2. BANCONTACT QR CODE Note: There is no impact on the Order API for this method. The payment_product parameter remains bancontactqrcode . #### S2S Notification Impact (Data Addition) Similar to the classic flow, notifications for statuses 116 (Authorized) and 118 (Captured) will now include the buyer's IBAN, BIC, and the 3DSecure block. Note that card expiration details (Month/Year) and PAN are generally not provided in the QR Code flow parameters. ## S2S NOTIFICATION PAYLOAD COMPARISON Below are real, anonymized examples of callbacks sent to your server.
